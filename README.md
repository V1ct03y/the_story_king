# 📢 Git 故事王 · 关键词生死局

**Git 流程验收项目**

> **流程必须规范，剧情可以发疯。**

---

## 活动目标

在不破坏团队分支规范的前提下，完成一场多人故事接龙，检验以下能力：

- 基于 `develop` 的 feature 开发流程
- PR 提交流程与描述质量
- 分支同步与冲突处理
- 团队协作与代码评审沟通

> Git 流程规范请严格遵守《类 GitFlow 开发流程说明》，本文不再重复。

---

## 故事玩法（《故事王》模式）

### 开场（固定）

首段剧情已提交：

> 峨眉山顶，清风道长闭关三十年终于悟出绝世剑法"天外飞仙"。出关那日，他白衣飘飘立于悬崖边，对前来挑战的魔教教主冷冷道："此剑一出，天地变色，你可想清楚了？"说罢缓缓拔剑——却发现剑鞘里插着的不是宝剑，而是一部正在响铃的 iPhone 14 Pro Max，来电显示："妈妈"。

第一棒必须解释：**为什么道长会有手机**（可穿越、可反转、可科幻，但要自洽）。

---

### 关键词抽卡机制

- 每回合作者抽取 **2~3 个关键词** 以及自己是**守序组**还是**混沌组**（从 `cards.exe` 关键词池中抽取）
- 当回合必须**全部使用**
- 关键词要**可检索、可定位**（不能"我心里用了"）
- 用词需自然融入剧情，不允许纯列表硬塞

---

### 阵营设定（仅剧情对抗）

- **守序组**：推进主线，确保世界观稳定
- **混沌组**：在剧情中制造反转 / 误导 / 离谱设定

> ⚠️ 只允许在故事里"捣乱"，**Git 流程不许乱**。

---

### 终局条件

全员必须最终达成预设结局（活动开始前公布一句"终局句子"）。

最后一位需发起**总收束 PR**到main分支，完成伏笔回收与关键词闭环。

---

## 回合流程

每位成员每回合执行以下操作：

1. 从最新 `develop` 切出 `feature/<name>-story-r<round>`
2. 在 `story.md` 中追加自己回合的剧情内容
3. 提交并 push 到远端
4. 发起 PR 到 `develop`
5. 若 `develop` 在你开发期间有更新，合并前先同步并处理冲突
6. PR 合并后，切回 `develop` pull 最新，删除本地 feature 分支

---

## PR 描述模板（必填）

每个 PR 必须包含以下内容：

```
## 剧情摘要
（本回合推进了什么）

## 关键词使用
- 关键词1：出现在第 X 段，"……原文引用……"
- 关键词2：出现在第 X 段，"……原文引用……"

## 埋坑点 / 钩子
（留给下一棒的线索或反转）

## 是否已同步最新 develop
- [ ] 是
```

---

## 评分标准（100 分）

| 维度 | 分值 | 说明 |
|------|------|------|
| 流程合规 | 25 | 分支策略、PR 路径、禁止直推 |
| 提交质量 | 20 | commit message、粒度、可读性 |
| 冲突处理 | 20 | 同步意识、解决质量、历史整洁 |
| 协作沟通 | 15 | review 质量与反馈闭环 |
| 剧情完成度 | 20 | 关键词自然度、逻辑自洽、终局闭环 |

**加分项：**
- 圆坑能力强（把他人离谱设定合理化）
- PR 描述清晰、review 有建设性意见
